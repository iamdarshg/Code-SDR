# Issue 12 FFT Receive Path

This revision implements the 1024-point receive FFT as a streaming path in the
105 MHz ADC clock domain. The FFT accepts one complex sample whenever
`data_valid` is asserted; it does not use a load/process/output batch schedule.

## Output And Ordering

`fft_processor` labels each valid output with `fft_index`. Tests must treat the
index as the authoritative natural bin number and verify that every frame emits
each index `0..1023` exactly once. Packet metadata carries frame count, bin
range, FFT size/mode, timestamp, packet sequence, and status flags.

## Overflow Metadata

Arithmetic overflow is reported as frame-local metadata and is expected to clear
between frames. Cocotb coverage includes a high-amplitude frame followed by a
quiet frame to catch sticky overflow regressions.

## Packet FIFO Fault Path

The FFT packet FIFO is intentionally monitored. If the Ethernet side cannot
drain bins quickly enough and the FIFO fills, the packetizer marks the
`dropped-bin` flag in metadata word 259. That flag is a fault indicator: packets
with `dropped-bin` set must not be treated as complete, lossless spectra by host
software. A clean issue #12 receive run requires this flag to remain clear.

The packetizer also checks subframe continuity and marks `discontinuity` in
metadata word 259 if the bins used for a packet are not a complete contiguous
subframe.

## UDP/IP Framing

The UDP checksum remains zero, which is legal for IPv4. The IPv4 header checksum
is generated by `udp_ip_stack` and covered by the UDP stack testbench.
